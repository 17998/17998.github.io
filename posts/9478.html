1.什么是Maven

Maven：

  1. 全新的自动化构建工具
  2. 一款开源的项目管理工具
  3. Maven仓库是基于简单文件系统存储的，集中化管理Java API资源（构件）的一个服务
  4. Maven工具是基于POM(Project Object Model,项目对象模型)模式实现的
  5. 在Maven中每一个项目都相当于是一个对象
  6. 仓库中的任何一个构件都有其唯一的坐标，根据这个坐标可以定义其在仓库中的唯一存储路径

目前技术在开发中存在的问题：

  1. 一个项目就是一个工程
  2. 项目中需要的jar包必须手动复制，粘贴到WEB-INF/lib目录下
  3. jar包需要别人替我们准备好，或去官网下载
  4. 一个jar包依赖的其他jar包需要自己手动添加到项目中

2.Maven仓库

2.1远程仓库

不在本机中的一切仓库都是远程仓库

默认的远程仓库使用Apache提供的中央仓库

远程仓库分为：

  • 中央仓库

  • 本地私服仓库

架设在局域网内的仓库服务，供局域网内的Maven用户使用

2.2本地仓库

指本机的一份拷贝,用来缓存远程下载，包含尚未发布的临时构件

2.2.1本地仓库配置

  1. 官网下载maven

    下载maven

  2. 配置Idea

  3. 配置settings

    1. 打开settings.xml

    2. 配置本地仓库的地址

      ctrl+f搜索localRepository，添加以下代码

      <!--本地仓库地址-->
<localRepository>D:\Maven\repository</localRepository>

    3. 配置阿里云镜像

      因为默认远程仓库在国外因此下载资源较慢

      ctrl+f搜索mirrors，将以下代码放入此标签内

      <mirror>
      <!--指定镜像ID-->
      <id>nexus-aliyun</id>
      <!--匹配中央仓库(阿里云的仓库，不可以自己起名)-->
      <mirrorOf>central</mirrorOf>
      <!--指定镜像名称(可以自己改名)-->
      <name>Nexus aliyun</name>
      <url>http://maven.aliyun.com/nexus/content/groups/public</url>
</mirror>

由近到远

3Maven工程

3.1POM工程

pom工程是一个逻辑工程。用在父级工程或聚合工程中。用来做jar包的版本控制

3.2JAR工程

打包成jar,用做jar包来使用。即常用的本地工程--->Java Project

3.3WAR工程

打包成war,发布在服务器上的工程

4Maven目录结构

  • src/main/java

    存储java源代码

  • src/main/resources

    存储主要的资源文件。比如xml配置文件或properties属性文件

  • src/test/java

    存储测试用的类，比如JUtil的测试一般就放在这个目录下面

    测试类本身是不属于项目的，所以maven专门创建一个测试包放

  • src/test/resources

    创建测试环境用的资源文件

  • src

    包含了项目所有的源代码和资源文件，以及其他项目相关的文件

  • target

    编译后内容放置的文件夹

  • pom.xml

    是maven的基础配置文件。配置项目和项目之间关系，包括配置依赖关系等

5Maven工程关系

创建项目后，选择IDEA最右侧的Maven面板lifecycle,双击install后(可以先点击clean)，就会把项目安装到本地仓库，其他项目就可以通过坐标引用此项目

5.1依赖关系

1.即A工程开发或运行过程中需要B工程提供支持，则代表A工程依赖B工程

在这种情况下，需要在A项目的pom.xml文件中增加下属配置定义依赖关系

通俗理解：就是导入jar包

B工程可以是自己的项目打包后的jar包，也可以是中央仓库的jar包

2.如何注入依赖

​ 在pom.xml文件中根元素project下的dependencies标签中，配置依赖信息，内可以包含多个dependence元素，以声明多个依赖。每个依赖dependence标签都应包含有以下元素：groupid,artifactid,version，这是依赖的基本坐标，对于任何一个依赖来说，基本坐标是最重要的，Maven根据坐标才能找到需要的依赖

<!--通过这个节点来添加依赖-->
    <dependencies>
        <dependency>
            <groupId>org.mybatis</groupId>
            <artifactId>mybatis</artifactId>
            <version>3.5.6</version>
        </dependency>
    </dependencies>

3.依赖的好处

  • 省去了程序员手动添加jar包的操作，省事

  • 可以帮助我们解决jar冲突问题

5.1.1依赖的传递性

假设你的项目依赖一个库，而这个库有依赖于其他库，但你不必去找出所有的这些依赖，你只需要加上你直接依赖的库，Maven会隐式的把这些库间接依赖也加入到你的项目中

如果A依赖了B，那么C依赖A是就会自动把A和B都导入进来

5.1.2依赖的原则

  • 第一原则：最短路径优先原则

    • 意味着项目依赖关系树种路径最短的版本会被使用

    • 假如A,B,C之间的依赖关系是A->B->C->D(2.0)和A->E->D(1.0),那么D(1.0)会被使用,因为A通过E到D的路径更短

  • 第二原则：最先声明原则

    依赖路径长度一样的时候，第一原则不能解决所有问题，A->B->Y(1.0),A->C->Y(2.0),Y(1.0)和Y(2.0)的依赖路径长度是一样的，都为2，在依赖路径长度相等的前提下，在POM中依赖声明的顺序决定了谁会被解析使用。使用顺序最靠前的依赖

5.1.3依赖范围

依赖范围决定了你依赖的坐标在什么情况下有效，什么情况下无效

  • compile

    默认范围。表示该依赖在编译和运行时够生效

    <dependency>
     <groupId>org.mybatis</groupId>
     <artifactId>mybatis</artifactId>
     <version>3.5.6</version>
     <scope>compile</scope>
</dependency>

  • provided

    已提供依赖范围。使用此依赖范围的Maven依赖。典型例子是servlet-api

  • runtime

    编译时不需要生效，而只在运行时生效。典型的例子是JDBC驱动

  • system

    与provided类似，不过你必须显示的指定一个本地系统路径的jar,maven也不会去仓库中找

    <dependency>
     <groupId>org.mybatis</groupId>
     <artifactId>mybatis</artifactId>
     <version>3.5.6</version>
     <scope>system</scope>
     <systemPath>本地jar路径</systemPath>
</dependency>

  • test

    只在编译测试和运行测试时需要。典型的例子就是JUtil

  • import

    只适用于pom中的部分。表名指定的POM必须使用部分的依赖

父工程(版本控制)

子工程

如果在父工程中写了<scope>import</scope>,那么子工程只能用父工程提供的版本

5.2继承关系

  • 如果A工程继承B工程，则代表A工程默认依赖B工程依赖的所有资源，且可以应用B工程中定义的左右资源信息
  • 被继承的工程(B工程)只能是POM工程
  • 在父项目中放在<dependencyManagement>中的内容不会被子项目继承，不可以直接使用，放在<dependencyManagement>中的内容主要是进行版本管理。里面的内容在子项目中依赖时坐标不需要写<version>，如果子项目不想使用父项目的版本可以自己明确给出version

本质上：POM文件的继承

5.3聚合关系

当我们开发的工程拥有2个以上模块的时候，每个模块都是一个独立的功能模块，开发的时候每个模块都可以独立编译，测试，运行。这个时候我们就需要一个聚合工程

  • 在创建聚合工程的过程中，总的工程必须是一个POM工程（jar项目，war项目是没有办法聚合的），子模块可以是任意类型的模块
  • 前提：继承

定义POM项目

<!--定义POM项目-->
    <packaging>pom</packaging>
<!--定义版本号-->
    <properties>
        <mybatis>3.5.6</mybatis>
        <spring>5.3.4</spring>
    </properties>
    <dependencyManagement>
        <dependencies>
            <dependency>
                <groupId>org.mybatis</groupId>
                <artifactId>${mybatis}</artifactId>
            </dependency>
            <dependency>
                <groupId>org.springframework</groupId>
                <artifactId>spring-context</artifactId>
                <version>${spring}</version>
            </dependency>
            <dependency>
                <groupId>org.springframework</groupId>
                <artifactId>spring-core</artifactId>
                <version>${spring}</version>
            </dependency>
        </dependencies>
    </dependencyManagement>

6.编译器插件

指定项目编译时使用的JDK和编码

  • 配置全局编译器插件

    • settings.xml文件中配置全局编译器插件
    • 找到profiles节点，在里面插入profile节点
    <profile>
    <!--定义的编译器插件id,全局唯一，名字随便起-->
    <id>jdk-1.7</id>
    <!--插件标记，activeByDefault:true默认编译器-->
    <activation>
        <activeByDefault>true</activeByDefault>
        <jdk>1.7</jdk>
    </activation>
    <!--配置信息source-源信息，target-字节码信息，compilerVersion-编译过程版本-->
    <properties>
        <maven.compiler.source>1.7</maven.compiler.source>
        <maven.compiler.target>1.7</maven.compiler.target>
        <maven.compiler.compilerVersion>1.7</maven.compiler.compilerVersion>
    </properties>
</profile>

  • 配置单个项目的编译器插件:pom.xml

    <build>
    <plugins>
        <!--JDK编译插件-->
        <plugin>
            <!--插件坐标-->
            <groupid>org.apache.maven.plugins</groupid>
            <actifactid>maven-compiler.plugin</actifactid>
            <version>3.2</version>

            <configuration>
                <!--源代码使用jdk版本-->
                <source>1.7</source>
                <!--源代码编译为class文件的版本，要跟上面保持一致-->
                <target>1.7</target>
                <encoding>UTF-8</encoding>
            </configuration>
        </plugin>
    </plugins>
</build>

7.资源拷贝插件

只有放在src/main/resources下咋配置文件才会被打包放入classes文件夹下面

我们现在向把非resources下面的文件也打包到classes下面

<build>
    <resources>
        <resource>
            <directory>src/main/java</directory>
            <includes>
                <include>**/*.xml</include>
            </includes>
        </resource>
        <resource>
            <directory>src/main/resources</directory>
            <includes>
                <include>**/*.xml</include>
                <include>**/*.properties</include>
            </includes>
        </resource>
    </resources>
</build>

8.Tomcat插件

过去

  • war项目必须部署在服务器上
  • 将Idea与外部Tomcat产生关联，然后部署到外部Tomcat上

现在

  • 不需要依赖外部的Tomcat，我们可以使用maven提供的Tomcat插件
  • 使用Tomcat插件发布部署并执行war工程，需要使用启动命令
  • 启动命令：tomcat7:run。tomcat7是插件命名
<!--配置tomcat插件-->
<plugin>
    <groupid>org.apache.tomcat.maven</groupid>
    <artifactid>tomcat7-maven-plugin</artifactid>
    <version>2.2</version>
    <configuration>
        <!--配置tomcat监听窗口-->
        <port>8080</port>
        <!--配置项目的访问路径-->
        <path>/</path>
    </configuration>
</plugin>

9.Maven常见命令

  • install

    本地安装，包含编译，打包，安装到本地仓库

    • 编译：javac
    • 打包：jar
    • 安装到本地仓库：将jar文件保存到本地仓库

  • clean

    清除已编译信息，删除工程中的target目录

  • compile

    只编译，javac命令

  • package

    打包。包含编译，打包两个功能
