    public class MyBeanPostProcessor implements BeanPostProcessor {

        return bean;

    }

@Transactional(rollbackFor = IOException.class)

  • readOnly：当事务中全是查询操作时， 才能将readOnly设置为true，否则会报错
  • timeout：超时回滚，释放资源，在规定时间内，事务没有执行完，强行回滚事务并抛出异常

1.24.5 事务隔离级别

  • 数据库系统必须具有隔离并发运行各个事务的能力，使他们不会互相影响，避免各种并发问题，一个事务与其他事务的隔离程度称为隔离级别，隔离级别越高，数据一致性就越好，但并发性越弱

  • 隔离级别一共有四种：

    • 读未提交：READ UNCOMMITTED

      • 允许事务A读取事务B未提交的事务

      • 可能出现脏读

        A读取到B未提交的数据，但B此时回滚了事务，导致A读取到了脏数据

    • 读已提交：READ COMMITTED

      • 允许事务A读取事务B未提交的事务

      • 可能出现不可重复读

        A先读取数据，B新增了数据，还未提交，此时A是读取不到新增的数据，当B提交了事务后，会导致A两次读取到的数据不一致

    • 可重复度：REPEATABLE READ

      • 确保事务A可以多次从一个字段中读取到相同的值，即事务A执行期间禁止其他事务对这个字段进行更新

    • 串行化：SERIALIZABLE

      • 确保事务A可以多次从一个表中读取到相同的行，在事务A执行期间，禁止其他事务对这个表进行添加，更新，删除操作，可以避免任何并发问题，但性能十分低下。

1.24.6 事务传播行为

  • 事务传播行为指的就是当一个事务方法被另一个事务方法调用时，这个事务方法应该如何进行。

  • transfer开启了一个事务，log也开启了一个事务，按理说log事务应该加入到transfer事务中，与transfer事务同成功同失败

  • 事务传播行为可以让事务协调员具有不同的事务行为，此处transfer调用log，log不是加入其事务，而是重新开启一个新的事务，不会与transfer事务同成功同失败

2.3 REST风格

2.3.1 简介

  • 使用这种形式来描述资源称为REST风格，使用这种风格来访问资源称为RESTful

2.3.2 入门案例

  • 一个是通过请求体传参，一个是通过路径传参
//RESTful
@Controller
public class UserController {
    @RequestMapping(value = "/users/{id}",method = RequestMethod.POST)
    @ResponseBody
    public String save(@PathVariable Integer id){
        System.out.println("save user:"+id);
        return "{model:save}";
    }

    @RequestMapping(value = "/users/{id}",method = RequestMethod.DELETE)
    @ResponseBody
    public String delete(@PathVariable Integer id){
        System.out.println("delete user:"+id);
        return "{model:delete}";
    }

    @RequestMapping(value = "/users/{id}",method = RequestMethod.PUT)
    @ResponseBody
    public String update(@PathVariable Integer id){
        System.out.println("update user:"+id);
        return "{model:update}";
    }

    @RequestMapping(value = "/users/{id}",method = RequestMethod.GET)
    @ResponseBody
    public String get(@PathVariable Integer id){
        System.out.println("get user:"+id);
        return "{model:get}";
    }
}

2.3.3 RESTful快速开发

  • 可以将公共部分@ResponseBody，和@RequestMapping(value=”/users”)提到类上
  • @RestController等价于@Controller+@ResponseBody

2.3.4 HiddenHttpMethodFilter处理PUT和DELETE请求

  • 浏览器只能发送GET和POST请求，若要发送PUT和DELETE请求需要在web.xml中配置一个过滤器HiddenHttpMethodFilter
  • 配置了过滤器之后，发送的请求必须满足两个条件才能将请求方式转为put和delete
    • 当前请求的请求方式必须是POST
    • 当前请求必须携带请求参数_method，其值才是最终的请求方式

2.7 SpringMVC执行流程

2.7.1 SpringMVC常用组件

  • DispatcherServlet：前端控制器，不需要工程师开发，由框架提供
    • 统一处理请求和响应，整个流程控制的中心，由它调用其他组件处理用户的请求
  • HandlerMapping：处理器映射器，不需要工程师开发，由框架提供
    • 根据请求的url,method等信息查找handler,即控制器方法
  • handler：处理器，需要工程师开发
    • 在DispatcherServlet的控制下handler对具体的用户请求进行处理
  • HandlerAdapter：处理器适配器，不需要工程师开发，由框架提供
    • 通过HandlerAdapter对处理器(控制器方法)进行执行
  • ViewResolver：视图解析器，不需要工程师开发，由框架提供
    • 进行视图解析，得到相应的视图，例如RedirectView，InternalResourceView(转发视图)
  • View：视图
    • 将模型数据通过页面展示给用户

2.7.2 DispatcherServlet初始化过程

  • DispatcherServlet本质上就是一个Servlet
  • 初始化时创建Web容器

2.7.3 SpringMVC执行流程

  1. 用户向服务器发送请求，请求被SpringMVC前端控制器DispatcherServlet捕获
  2. DispatcherServlet对请求URL(资源在网络中的路径)进行解析，得到请求资源标识符URI(资源在服务器中的路径)，判断请求URL对应的映射
    • 不存在(DispatcherServlet处理不了)
      • 再判断是否配置了mvc:default-servlet-handler(默认servlet)
        • 如果没有配置，则控制台报映射查找不到，客户端展示404
        • 如果有配置，则访问目标资源(一般为静态资源)，找不到客户端也会展示404
    • 存在则执行下面的流程
      • 根据该URL，调用HandlerMapping获得该handler配置的所有相关的对象(包括Handler对象，以及Handler对象对应的拦截器)，最后以HandlerExecutionChain执行链对象的形式返回
      • DispatcherServlet根据获得的Handler,选择一个合适的HandlerAdapter
      • 如果成功获得HandlerAdapter，此时将开始执行拦截器的preHandler方法
      • 提取Request中的模型数据，填充Handler入参，开始执行Handler(Controller方法)，处理请求，在填充Handler入参过程中，根据配置，Spring会做一些额外工作
        • HttpMessageConveter：将请求消息(JSON)转换成一个对象，将对象转换为指定的响应信息
        • 数据转换：对请求消息进行数据转换，如：String转为Integer
        • 数据格式化，对请求消息进行数据格式化，如：格式化日期
        • 数据验证：验证数据的有效性
      • Handler执行完成后，向DispatcherServlet返回一个ModelAndView对象
      • 此时将开始执行拦截器的PostHandler方法
      • 根据返回的ModelAndView对象(此时会判断是否抛出异常，如果抛出异常，则执行HandlerExceptionResolver进行异常处理)选择一个合适的视图解析器ViewResolver进行视图解析，根据Model和View来渲染视图
      • 将渲染结果返回给客户端

第三部分 Mybatis

3.1 Mybatis特性

  1. Mybatis支持定制化SQL,存储过程以及高级映射的优秀的持久层框架
  2. Mybatis避免了几乎所有的JDBC代码和手动设置参数以及获取结果集
  3. Mybatis可以使用简单的XML或者注解用于配置和原始映射，将接口和Java的POJO(Plain Old Java Objects)映射成数据中的记录
  4. Mybatis是一个半自动的ORM(Object Relation Mapping)框架

3.2 Mybatis获取参数的各种情况

${}：本质是字符串的拼接

#{}：本质是占位符赋值

3.2.1 mapper接口方法的参数为单个的字面量类型

public interface EmployeeMapper {
    /**
     * 查询员工详细信息
     * @param id
     * @return
     */
    EmployeeVo selectEmployeeAndRoleById(Integer id);
}

<?xml version="1.0" encoding="UTF-8" ?>
<!DOCTYPE mapper
        PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
        "https://mybatis.org/dtd/mybatis-3-mapper.dtd">

<mapper namespace="com.gxa.mapper.EmployeeMapper">
    <resultMap id="empMap" type="EmployeeVo">
        <id property="id" column="empl_id"/>
        <result property="name" column="empl_name"/>
        <result property="type" column="empl_type"/>
        <collection property="roles" ofType="Role">
            <id property="id" column="role_id"/>
            <result property="name" column="role_name"/>
            <result property="remark" column="role_remark"/>
        </collection>
    </resultMap>

    <select id="selectEmployeeAndRoleById" resultMap="empMap" parameterType="integer">
        select e.empl_id,e.empl_name,e.empl_type,r.role_id,r.role_name,r.role_remark from tb_employee e left join empl_role er on e.empl_id = er.empl_id
        left join tb_role r on er.role_id = r.role_id where e.empl_id = #{id}
    </select>
</mapper>

 ${}和#{}中可以是任意的字符串

    • 以arg0，arg1…..为键，参数为值
    • 以param1,param2….为键，参数为值

  • 通过@Param注解指定一个值，将该值作为Map的key，参数作为Map的value

3.4 Mybatis缓存

3.4.1 Mybatis一级缓存

  • 一级缓存是SqlSession级别的，通过同一个SqlSession查询的数据会被缓存下来，下次查询相同的数据就会直接从缓存中获取，不会访问数据库
  • 一级缓存默认是开启的，即使是同一类型的不同Mapper实例来执行同样的查询操作仍然会缓存下来

3.4.2 一级缓存失效情况

  1. 不同的SqlSession对应不同的一级缓存
  2. 同一个SqlSession但是对应的查询条件不一样
  3. 同一个SqlSession两次查询期间进行了任何的一次增删改操作
  4. 同一个SqlSession两次查询期间手动清空了缓存

3.4.3 MyBatis二级缓存

  • 二级缓存是SqlSessionFactory级别的，通过同一个SqlSessionFactory创建的SqlSession查询的结果会被缓存，此后若再次执行相同的查询语句，结果就会从缓存中获取
  • 开启二级缓存的条件
    1. 在核心配置文件中，设置全局配置属性cacheEnabled=’true’,默认为true，不需要配置
    2. 在映射文件中设置标签
    3. 二级缓存必须在SqLSession关闭或提交之后有效
    4. 查询的数据所转换的实体类类型必须实现序列化接口
  • 二级缓存失效的情况
    1. 在两次查询过程中执行了任意的增删改，会使一级缓存和二级缓存失效

3.4.4 二级缓存的相关配置

3.4.5 MyBatis缓存查询顺序

  • 先查询二级缓存，因为二级缓存可能会有其他程序已经查出来的数据，可以拿来直接使用
  • 如果二级缓存没有命中，再查询一级缓存
  • 如果一级缓存也没有命中，再访问数据库
  • SqlSession关闭之后，一级缓存会写入二级缓存

3.5 MyBatis逆向工程

3.5.1 什么是逆向工程

  • 正向工程：先创建Java实体类，由框架负责根据实体类生成数据库表，Hibernate是支持正向工程的
  • 逆向工程：先创建数据库表，由框架负责根据数据库表，反向生成如下资源
    • Java实体类
    • Mapper接口
    • Mapper映射文件
